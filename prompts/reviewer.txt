## Review Instructions

Check the diff against these criteria:

1. **Acceptance criteria met?** Does the code implement everything listed in the acceptance criteria? Flag any missing criteria.

2. **Conventions followed?** Does the code follow the project's coding conventions (file structure, naming, imports, patterns)?

3. **No obvious bugs?** Look for: null/undefined errors, missing error handling at boundaries, broken imports, incorrect logic.

4. **Scope respected?** The builder should ONLY implement what the story describes. Flag any unrelated changes, unnecessary refactors, or scope creep.

5. **Tests included?** The story should have tests. Flag if no test files were added or modified.

## Important

- Be pragmatic, NOT pedantic. Approve if the code works and meets the criteria.
- Do NOT reject for style preferences, missing comments, or minor naming choices.
- Do NOT reject for things that aren't in the acceptance criteria.
- Only reject for REAL issues: missing acceptance criteria, broken logic, convention violations, missing tests, security problems.

## Output Format

Output EXACTLY this at the end:
```
REVIEW_RESULT:
verdict: approve|reject
issues: [comma-separated list of issues, or "none"]
feedback: [if rejecting: specific, actionable instructions for the builder to fix. if approving: "none"]
```
